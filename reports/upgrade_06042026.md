## Project Understanding

**Inglish** is a hybrid English → Hindi/Marathi code-mixed translation framework for Indian technical audiences (students, developers). Its three-tier pipeline achieves the key goal of preserving technical terminology:

1. **Tier 1 — Term Extraction & Guarding** (`term_extractor.py`): Trie-based compound-term lookup + single-term set → wraps matched terms in `[square brackets]`
2. **Tier 2 — LLM Translation** (`translator.py` + `llm_adapter.py`): LLM receives bracketed text and must keep brackets intact; returns JSON with `roman` and `devanagari` keys
3. **Tier 3 — Script Conversion** (`script_converter.py`): Word-level Roman→Devanagari map with sentinel-based 3-pass algorithm (currently unused by pipeline — LLM handles script output via prompt)

Supports 6 LLM providers (Gemini, Groq, OpenAI, Anthropic, Ollama, LM Studio) via a unified adapter with exponential backoff. Domain glossaries are YAML files — adding a domain requires no code changes.

## 🚨 Critical Issues

### 1. Tests are completely broken — `src/test_pipeline.py`

**Severity:** High — `pytest` will fail on every single test.

All test fixtures call:
```python
TranslationConfig(translator_type="baseline", output_format="both")
```
`TranslationConfig` has no `translator_type` or `output_format` fields — both were removed in a prior refactor. The `setup_method` crashes before any test body runs.

Additionally:
- `test_quality_metrics` calls `self.pipeline.evaluate_quality(...)` — this method does not exist on `InglishtranslationPipeline`.
- `TestTranslationConfig.test_default_config` asserts `config.translator_type == "baseline"` — field doesn't exist.
- The `sys.path.insert` at line 7 adds `parent.parent / "src"` — but the file is already *inside* `src/`, so the path is wrong.

**Action required:** Rewrite tests to match the current `TranslationConfig` and `InglishtranslationPipeline` API.

### 2. Baseline benchmark crashes before translating — `src/baseline_benchmark.py:91-95`

**Severity:** High — the benchmark is completely unusable without a live API key.

```python
config = TranslationConfig(domain=domain, target_language="hi", llm_provider="gemini")
pipeline = InglishtranslationPipeline(config)  # ← raises EnvironmentError here
```

`InglishtranslationPipeline.__init__` always instantiates `LLMTranslator` → `LLMAdapter` → `_init_gemini()` → reads `GEMINI_API_KEY` → raises `EnvironmentError` if not set. The inline comment *"will not be called — no API key for baseline"* is incorrect: the client is created eagerly at init, not lazily at call time.

The stated purpose of this benchmark is to measure the **rule-based baseline without any LLM** — it should be runnable with zero API keys.

**Action required:** Add an `llm_provider="none"` path to `LLMAdapter`/`InglishtranslationPipeline` that skips LLM initialisation, or pass a guard into the pipeline for baseline mode.

### 3. Compound term spans include trailing punctuation — `src/term_extractor.py:122-129`

**Severity:** Medium — corrupts term guarding for sentence-final compound terms.

In `_match_compound_terms`, the matched words list stores **original tokens** (with punctuation). If a compound term appears at the end of a sentence:

```
text     = "The for loop."
words    = ["The", "for", "loop."]
matched_words = ["for", "loop."]
match_text    = "for loop."   # period included!
```

The span end = `start + len("for loop.")` — the bracket wraps `[for loop.]`, which:
- Passes the period inside the bracket to the LLM
- Breaks `validate_constraints()` (the constraint checker looks for `[for loop]`, not `[for loop.]`)

**Action required:** Strip `_PUNCT` from the last matched word before computing `match_text` and `end`.

### 4. `resolve_overlapping_spans` doesn't guarantee longest-match-wins globally — `src/utils.py:67-86`

**Severity:** Low-Medium — incorrect output for certain input patterns; contradicts the stated contract.

The docstring says *"the longer one wins"*. The current algorithm:
1. Sorts spans by `(start, -length)` — so at the **same start position**, longer wins.
2. Iterates and greedily keeps spans that don't overlap with already-kept spans.

But: if span A starts at position 0 with length 5, and span B starts at position 3 with length 20, A is processed first and kept. B is then blocked — even though B is much longer. The longest-match-wins guarantee only holds for same-start ties.

**Example trigger:** Glossary has both `"loop"` (single term) and `"infinite loop"` (compound). If "loop" is matched at position 9 (length 4) and "infinite loop" is matched at position 0 (length 13), the current code would keep both because they don't overlap (positions 9 and 0 don't overlap with each other). However if single-term matching returns "loop" at position 9 and compound matching returns "infinite loop" at position 0, that's fine. The real risk is a single-term match at position 2 blocking a compound that starts at position 0 and ends at position 10.

**Action required:** Change to an interval-scheduling approach: sort all candidates by length descending, greedily pick longest non-overlapping.

## ♻️ Redundancy Findings

### 1. `unguard_terms` / `remove_brackets` defined three times identically

The same one-liner — `re.sub(r'\[([^\]]+)\]', r'\1', text)` — exists as:
- `BaseTranslator.unguard_terms` (`translator.py:43`)
- `TermExtractor.unguard_terms` (`term_extractor.py:96`)
- `remove_brackets` (`utils.py:53`)

`utils.py` is the shared utilities module. The other two should call `utils.remove_brackets` instead of reimplementing it.

### 2. `get_guarded_terms` duplicates `extract_bracketed_terms`

- `TermExtractor.get_guarded_terms` (`term_extractor.py:100`): `re.findall(r'\[([^\]]+)\]', text)`
- `utils.extract_bracketed_terms` (`utils.py:48`): identical

`TermExtractor` should delegate to `utils.extract_bracketed_terms`.

### 3. `backup/` folder is a committed copy of all source files

`backup/` contains verbatim copies of every file in `src/`. Git history is authoritative for this purpose. Keeping this folder in the repo creates drift risk (backup files diverge silently from src) and clutters navigation.

### 4. `ScriptConverter.generate_bilingual_output` is dead code (`script_converter.py:185-196`)

The pipeline never imports `ScriptConverter`. This method is unreachable in any current execution path.

## 📚 Documentation Gaps

### 1. `src/data/glossaries/` has no README

The YAML schema is not documented inline in a discoverable way. Contributors wanting to add a domain have to infer the format from existing files. A short README explaining `terms` vs `compound_terms`, what belongs in each, and why verbs are excluded would prevent common mistakes.

### 2. README.md roadmap shows physics/finance glossaries as incomplete — they already exist

```markdown
- [ ] Physics, Finance, Medicine domain glossaries   ← physics.yaml and finance.yaml exist
```

Both YAML files are present in `src/data/glossaries/`. The roadmap should be updated.

### 3. `backup/Notes_Inglish_Technical_Translation_Framework.md` is buried in the wrong place

Design notes that describe architectural rationale belong in `docs/` or the project wiki, not in a backup folder committed alongside code copies.
